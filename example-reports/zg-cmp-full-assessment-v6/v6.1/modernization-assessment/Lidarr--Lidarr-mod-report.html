Executive Summary

Repository Status: 🟠 Remediation Required

Key Findings: This music collection manager is a self-hosted .NET 8.0 monolith with no cloud deployment model. It has mature application code (.NET 8.0, comprehensive test suite, clean data access patterns) but lacks all cloud infrastructure: no IaC, no containers, no managed compute, no network security, and no observability beyond NLog and Sentry.

Remediation Plan: Phase 1 focuses on establishing cloud foundations (containerize, create IaC, add deployment pipeline). Phase 2 addresses security and observability gaps. Phase 3 handles architecture evolution (decomposition, service discovery).

Recommended Actions:

  1. Create a Dockerfile and deploy to Amazon EKS with Aurora PostgreSQL
  2. Define IaC (CDK/Terraform) for complete AWS infrastructure
  3. Add deployment stages to CI/CD pipeline (build → ECR → EKS)

This repo has 9 High findings, 8 Medium findings, 3 Low findings. Rule: 2-11 High → Remediation Required. Note: MOD "1 High" maps to Pilot-Ready (a single modernization gap), unlike ARA where 1 High is a deployment blocker.

Detailed Findings

HIGH INF-Q1: No Managed Compute

Category: Infrastructure & DevOps
Finding

All compute is bare-metal/VM-based. Lidarr is distributed as self-contained executables and OS-specific packages with no managed compute services (ECS, EKS, Lambda, Fargate).

Recommendation

Containerize the application using .NET 8.0 native container support and deploy to Amazon EKS. The environment variable configuration pattern and configurable port binding make it container-ready.

HIGH INF-Q2: Self-Managed Databases

Category: Infrastructure & DevOps
Finding

Databases are entirely self-managed. SQLite is the default; PostgreSQL is optional. Both run on same host or user-managed infrastructure with no managed service.

Recommendation

Migrate PostgreSQL workloads to Amazon Aurora PostgreSQL. Aurora provides automated backups, Multi-AZ failover, and read replica auto-scaling.

HIGH INF-Q5: No Network Security

Category: Infrastructure & DevOps
Finding

No VPC, security groups, NACLs, or network segmentation defined. Network security relies entirely on host OS firewall configuration.

Recommendation

Provision within a VPC with private subnets. Place application behind ALB/API Gateway with database in private subnets using least-privilege security groups.

HIGH INF-Q10: No Infrastructure as Code

Category: Infrastructure & DevOps
Finding

No IaC exists. No Terraform, CloudFormation, CDK, Helm charts, or Kubernetes manifests found.

Recommendation

Create IaC using CDK or Terraform to define AWS infrastructure: EKS cluster, Aurora PostgreSQL, ECR, VPC networking, ALB/API Gateway.

HIGH SEC-Q4: No Centralized Identity

Category: Security Baseline
Finding

Application manages its own authentication entirely with no external IdP integration. No OIDC, SAML, or federated identity.

Recommendation

Integrate with Amazon Cognito as the identity provider. Implement OIDC authentication flow for web UI and token-based auth for API.

HIGH SEC-Q6: No Compute Hardening

Category: Security Baseline
Finding

No compute hardening or patching strategy beyond application self-update. No vulnerability scanning, no hardened images.

Recommendation

Use hardened base images (Bottlerocket for EKS, distroless for containers). Enable ECR scanning. Integrate vulnerability scanning in CI/CD.

HIGH INF-Q6: No API Entry Point

Category: Infrastructure & DevOps
Finding

Services exposed directly via Kestrel with no gateway, load balancer, or CDN. No throttling, no WAF, no managed TLS termination.

Recommendation

Deploy behind Amazon API Gateway for REST API with throttling, validation, and auth. Use CloudFront for frontend static assets.

HIGH INF-Q7: No Auto-Scaling

Category: Infrastructure & DevOps
Finding

No auto-scaling mechanisms configured. Application runs as single-instance process with no scaling capability.

Recommendation

When deployed to EKS, configure Horizontal Pod Autoscaler (HPA) based on CPU/memory and request rate.

HIGH INF-Q8: No Automated Backups

Category: Infrastructure & DevOps
Finding

No automated backup configuration. SQLite on local filesystem. Manual backup feature exists but is not scheduled.

Recommendation

Leverage Aurora automated backups with PITR. Configure AWS Backup for cross-region replication.

MEDIUM INF-Q11: CI/CD Without Deployment

Category: Infrastructure & DevOps
Finding

Azure Pipelines provides comprehensive build and test but no deployment stage. Artifacts produced but not deployed automatically.

Recommendation

Add deployment stages: build container → push to ECR → deploy to EKS staging → smoke tests → promote to production.

MEDIUM APP-Q2: Monolith

Category: Application Architecture
Finding

Single deployable unit with 15 projects but shared database access and cross-module coupling. Internal event aggregator provides some decoupling.

Recommendation

Begin with Conditional/Adaptive approach: containerize as-is, then extract high-value services (notifications, downloads, metadata).

MEDIUM APP-Q3: Primarily Synchronous

Category: Application Architecture
Finding

Communication primarily synchronous. Internal event aggregator is fire-and-forget but in-process only. External calls all sync HTTP.

Recommendation

Adopt EventBridge for cross-service state changes when decomposing. Use SQS for durable work queues.

MEDIUM SEC-Q5: No Secrets Management

Category: Security Baseline
Finding

No plaintext credentials in source, but production credentials in environment variables without rotation or centralized management.

Recommendation

Store all credentials in AWS Secrets Manager with automated rotation. Use IAM roles for service-to-service authentication.

MEDIUM DATA-Q1: Local Filesystem Storage

Category: Data Platform
Finding

Music files, cover art, and metadata stored on local filesystems with no cloud object storage or CDN.

Recommendation

Store cover art in S3 with CloudFront CDN. Consider S3 File Gateway for media file access.

MEDIUM OPS-Q1: No Distributed Tracing

Category: Operations & Observability
Finding

No distributed tracing. NLog and Sentry only. No trace ID propagation or OpenTelemetry.

Recommendation

Integrate OpenTelemetry .NET SDK. Export to AWS X-Ray. Add W3C TraceContext headers.

LOW DATA-Q2: Mostly Centralized Data Access

Category: Data Platform
Finding

Data access mostly centralized through BasicRepository pattern. Some direct Dapper access for complex queries.

Recommendation

Maintain existing pattern. Connection string update is the primary change for Aurora migration.

LOW DATA-Q3: Database Versions Partially Managed

Category: Data Platform
Finding

CI tests against PostgreSQL 14/15. No explicit version pin in deployment. No documented upgrade procedure.

Recommendation

Pin engine version in IaC. Plan PG 14 EOL migration (Nov 2026).

LOW OPS-Q6: Integration Tests With Gaps

Category: Operations & Observability
Finding

Integration tests exist in CI against PostgreSQL in Docker. Automation tests provide E2E coverage. Minor coverage gaps.

Recommendation

Expand coverage to all critical API workflows. Add contract tests for third-party consumers.
