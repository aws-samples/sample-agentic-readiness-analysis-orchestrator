Detailed Findings

SEC-Q1: Audit Logging HIGH

Category: Security Baseline
Finding

No CloudTrail configuration or audit logging infrastructure is generated by any Lift construct. The plugin does not provision CloudTrail trails, CloudWatch log groups for audit purposes, or any logging infrastructure as part of its constructs.

Recommendation

Add an optional audit/logging configuration to the plugin's top-level lift config that provisions a CloudTrail trail covering all Lift-generated resources. Alternatively, document the audit logging requirements clearly and provide a recommended CloudTrail configuration example.

APP-Q5: API Versioning Strategy MEDIUM

Category: Application Architecture
Finding

No explicit versioning strategy for the construct schemas (the public API surface). Breaking changes to CONSTRUCTS_DEFINITION or individual construct schemas have no migration path or deprecation mechanism.

Recommendation

Introduce construct schema versioning (e.g., type: "queue/v2") or implement a deprecation-warning mechanism. Add a CHANGELOG.md with explicit breaking-change callouts per release.

DATA-Q4: Stored Procedures and Schema Complexity MEDIUM

Category: Data Platform
Finding

The DatabaseDynamoDBSingleTable construct hard-codes a specific single-table design pattern with fixed attribute names (PK, SK), up to 5 LSIs with hard-coded sort keys, and a hard-coded TimeToLive attribute name.

Recommendation

Make the DynamoDB construct's attribute names configurable via the schema definition. Allow consumers to specify custom partition/sort key names, TTL attribute name, and LSI/GSI sort key patterns.

OPS-Q1: Distributed Tracing MEDIUM

Category: Operations & Observability
Finding

The library does not instrument distributed tracing in any generated construct. No X-Ray or OpenTelemetry configuration is included in Lambda or API Gateway resources generated by the constructs.

Recommendation

Add optional tracing configuration (e.g., lift.tracing: true) that enables X-Ray active tracing on all generated Lambda functions and API Gateway stages.

SEC-Q4: Centralized Identity Integration MEDIUM

Category: Security Baseline
Finding

The library does not generate or integrate with any centralized identity provider (Cognito, Okta). No Cognito user pool construct exists. Website constructs do not offer built-in auth integration.

Recommendation

Consider adding a Cognito-based authentication construct (e.g., type: "auth") that provisions a Cognito User Pool and integrates with API Gateway authorizers and CloudFront signed URLs.

SEC-Q5: Secrets Management MEDIUM

Category: Security Baseline
Finding

The Stripe provider reads a secretKey from configuration without enforcing use of Secrets Manager or SSM Parameter Store. No runtime warning when secrets appear to be plaintext values.

Recommendation

Add documentation and validation guiding secrets through variable resolvers. Consider a runtime warning when secret-like configuration values do not use variable references.

APP-Q1: Programming Languages LOW

Category: Application Architecture
Finding

TypeScript 4.3 targeting ES2019 with Node.js 16 in .nvmrc. CI tests on modern Node versions (20/22/24). aws-sdk v2 used in devDependencies for testing.

Recommendation

Upgrade TypeScript to 5.x. Update .nvmrc to Node 20 LTS. Migrate test code from aws-sdk v2 to @aws-sdk v3 client packages.

APP-Q2: Monolith vs Microservices LOW

Category: Application Architecture
Finding

Well-structured modular architecture with clear separation (providers, constructs, utils). Plugin entry point handles multiple concerns but coupling is manageable for library size.

Recommendation

Current structure is appropriate. Consider extracting variable resolution into a dedicated module if the plugin grows significantly.
