### RISK-QUALITY — Address as Capacity Allows

#### DISC-Q1: Schema Versioning and API Contracts — RISK-QUALITY

- **Severity**: RISK-QUALITY
- **Finding**: The library follows semver (version 6.2.2 in `package.json`), exports TypeScript type definitions (`dist/index.d.ts`, `dist/index.d.mts`) generated by tsup, enforces strict TypeScript compilation (`tsconfig.json`: `strict: true`), and runs `tsd` type tests (configured in `package.json`: `"tsd": {"directory": "test"}`). The README.md documents all 5 public functions with parameter tables. However, the CI pipeline (`.github/workflows/ci.yml`) runs lint, build, and test only — there is **no automated breaking-change detection** (no Pact, no OpenAPI diff, no `tsd` diff, no explicit contract testing step beyond basic `tsd` type-checking). No CHANGELOG file was found in the repository.
- **Gap**: No automated breaking-change detection in CI pipeline. No CHANGELOG file to track API changes across versions. While `tsd` validates type definitions compile correctly, it does not detect breaking changes between versions (e.g., removed exports, changed function signatures).
- **Compensating Controls**:
  - The `tsd` type tests provide partial protection — if a type definition changes in a way that breaks the test assertions, CI will fail.
  - Semver compliance provides a social contract that breaking changes increment the major version.
- **Remediation Timeline**: 30–60 days
- **Recommendation**: Add a breaking-change detection step to CI (e.g., `@arethetypeswrong/cli`, `tsd` snapshot comparison, or a changelog enforcement tool like `changeset`). Maintain a CHANGELOG.md to document API changes.
- **Evidence**: `package.json` (version, tsd config), `.github/workflows/ci.yml` (CI pipeline), `tsconfig.json` (strict mode), `dist/index.d.ts` (type declarations), `README.md` (API documentation)
