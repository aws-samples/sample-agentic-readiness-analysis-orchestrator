## Executive Dashboard

This portfolio of four e-commerce services scores **1.66 / 4.0** overall — placing the entire portfolio firmly in the "Needs Work" category for enabling an AI agent that handles order status inquiries, processes returns, and manages inventory restocking. No service in the portfolio is agent-ready today; three services (local-monolith, aws-microservices, books-api) need significant foundational work, and one (unishop-monolith) is critically unprepared with no IaC, no CI/CD, no authentication, and no containerization.

The strongest area across the portfolio is structured JSON API responses — all four services return well-formed JSON that an agent can parse immediately. The aws-microservices repo provides a solid event-driven microservices pattern (EventBridge + SQS + DynamoDB) that demonstrates the target architecture. The books-api has the most mature CI/CD pipeline with linear deployments and automated rollback. However, every service shares the same critical blockers: **zero API documentation** (the agent cannot discover tools), **zero AI/agent frameworks** (no Bedrock, Strands, or LangChain integration exists), **zero vector databases or RAG pipelines** (no semantic search for customer inquiries), and **near-zero observability** (no structured logging, no SLOs, no anomaly detection).

The recommended approach is a 6-month phased modernization starting with cross-cutting foundations (CI/CD, observability, API documentation across all services), followed by agent quick wins leveraging existing JSON APIs, then building the full agent infrastructure (RAG, eval pipelines, human approval workflows), and finally scaling to production-grade agent operations. The circular dependency between aws-microservices and books-api must be resolved in Phase 0 before independent modernization can proceed. A centralized platform team is recommended to drive shared infrastructure (Cognito, EventBridge, observability stack) while service teams focus on domain-specific agent tool development.

## Service Dependency Map

### High-Level Architecture

The portfolio consists of four e-commerce services with varying maturity levels. Two monolithic applications (local-monolith on App Runner, unishop-monolith on EC2) handle the core order management and inventory domains. One microservices platform (aws-microservices on Lambda) provides event-driven product, basket, and ordering capabilities. One serverless API (books-api on Lambda) provides book catalog operations with the most mature CI/CD pipeline.

A circular dependency exists between aws-microservices and books-api: the microservices ordering flow triggers book catalog updates via EventBridge events (async), while the books-api queries the product microservice for catalog data via REST (sync). The two monoliths operate independently with no direct service-to-service dependencies.

### Critical Path Analysis

1. **Circular Dependency (Phase 0 — must resolve first):**
   - aws-microservices ↔ books-api: Both sync and async dependencies create a cycle. This blocks independent deployment and modernization. Resolution: Make the EventBridge communication unidirectional (aws-microservices publishes, books-api subscribes only) and replace the sync REST call from books-api with an EventBridge query or cached local copy.

2. **Isolated Services (can be parallelized):**
   - local-monolith and unishop-monolith have no service dependencies and can be modernized concurrently with the coupled pair.

3. **Independent Services (can start immediately):**
   - All 4 services can begin Phase 1 activities (CI/CD, API docs, structured logging) independently after Phase 0 cross-cutting foundations are established.

### 🚨 Foundational Blockers

> These gaps block all modernization efforts, not just `enable-agentic-use-case`.
> Address these first — nothing else matters until these are resolved.

1. **INF-Q6: CI/CD Pipelines** — 3 of 4 services score < 2
   - **Impact**: Without CI/CD, no automated testing, no repeatable deployments, no rollback capability. Every deployment is a manual, error-prone process. Agent development requires rapid iteration cycles that manual deployments cannot support.
   - **Affected services**: local-monolith (1/4), unishop-monolith (1/4), aws-microservices (1/4)
   - **Recommendation**: Establish CI/CD pipelines for all 3 services using GitHub Actions or AWS CodePipeline. Use books-api's pipeline (CDK + CodePipeline with linear deployments) as the reference implementation. Standardize on a shared pipeline template.

2. **OPS-Q1: Distributed Tracing** — 3 of 4 services score < 2
   - **Impact**: Without tracing, agent workflows spanning multiple services are invisible. When an agent calls the order API, then the inventory API, then processes a return — and something fails — there is no way to reconstruct the execution path.
   - **Affected services**: local-monolith (1/4), unishop-monolith (1/4), aws-microservices (1/4)
   - **Recommendation**: Enable AWS X-Ray or OpenTelemetry across all services. books-api already has X-Ray enabled — replicate this pattern. For the agent layer, add `gen_ai.*` semantic conventions for LLM call tracing.

3. **APP-Q8 + SEC-Q5: Rate Limiting & API Rate Limits** — 4 of 4 services score < 2 on both criteria
   - **Impact**: No rate limiting at any layer across the entire portfolio. A misconfigured agent in a retry loop could send unlimited requests, overwhelming backends, incurring unbounded costs, and causing cascading failures.
   - **Affected services**: local-monolith (1/4), unishop-monolith (1/4), aws-microservices (1/4), books-api (1/4)
   - **Recommendation**: Deploy API Gateway with usage plans and throttling across all services. Create dedicated API keys for the agent service with appropriate rate limits. Add WAF rate-based rules as secondary defense.

### ⚠️ Prerequisites for `enable-agentic-use-case`

> These gaps specifically block your path to `enable-agentic-use-case`.
> They aren't the goal itself, but you can't get there without them.

1. **APP-Q2: API Documentation** — 4 of 4 services score < 3
   - **Impact on goal**: Agents need machine-readable API specifications (OpenAPI/Swagger) to discover and invoke service endpoints as tools. Without API documentation, the agent for order status inquiries, returns, and inventory restocking literally cannot know what endpoints exist, what parameters they accept, or what responses to expect.
   - **Affected services**: local-monolith (1/4), unishop-monolith (1/4), aws-microservices (1/4), books-api (1/4)
   - **Recommendation**: Launch a portfolio-wide API documentation sprint. Generate OpenAPI 3.0 specs for all services, prioritizing agent-critical endpoints: order status, return submission, inventory lookup, basket operations, and catalog queries. Use these specs to define agent tools.

2. **SEC-Q3: Identity Propagation** — 4 of 4 services score < 3
   - **Impact on goal**: The agent must act on behalf of specific customers — "check MY order status", "process MY return". Without identity propagation, the agent cannot securely associate actions with authenticated users, creating both security and compliance risks.
   - **Affected services**: local-monolith (1/4 — PHP sessions), unishop-monolith (1/4 — OAuth2 disabled), aws-microservices (1/4 — username as path param), books-api (2/4 — Cognito on POST only)
   - **Recommendation**: Deploy Amazon Cognito as a centralized identity provider. Implement JWT-based authentication across all services. Issue machine-to-machine credentials for the agent via OAuth2 client credentials flow. Propagate user identity through JWT tokens so agent actions are attributed to the correct customer.

## Integration Opportunities

### Shared Service Extraction

**Opportunity 1: Centralized Authentication (Cognito)**
- **Current State**: 4 different auth patterns — PHP sessions (local-monolith), disabled OAuth2 (unishop-monolith), no auth (aws-microservices), partial Cognito (books-api)
- **Proposed Solution**: Deploy a shared Amazon Cognito User Pool with machine-to-machine credentials for the agent and user-facing OAuth2 flows
- **Benefits**: Single identity provider, consistent auth across all services, agent can use client credentials flow
- **Effort**: Medium
- **Priority**: High — prerequisite for secure agent deployment

**Opportunity 2: Unified Observability Platform**
- **Current State**: 3 services have no tracing, 4 services have no structured logging, only books-api has X-Ray
- **Proposed Solution**: Deploy shared observability stack — X-Ray/OpenTelemetry for tracing, Lambda Powertools for logging, CloudWatch dashboards for metrics, with shared templates and standards
- **Benefits**: End-to-end agent workflow tracing, consistent log format, centralized dashboards
- **Effort**: Medium
- **Priority**: High — required for agent debugging and monitoring

**Opportunity 3: Agent Orchestration Layer**
- **Current State**: No agent exists in any service. Each service has independent APIs.
- **Proposed Solution**: Build one centralized agent orchestrator service on ECS that calls all 4 backend services as tools via their OpenAPI-defined APIs
- **Benefits**: Single agent with multi-service capabilities, centralized prompt management, unified eval pipeline
- **Effort**: High
- **Priority**: High — the core deliverable of the agentic use case

### Event-Driven Architecture

**Opportunity 1: Portfolio-Wide EventBridge**
- **Current State**: Only aws-microservices uses EventBridge (SwnEventBus). The rest have 100% synchronous communication.
- **Proposed Solution**: Extend the SwnEventBus as a portfolio-wide event bus. Publish domain events from all services (order-created, return-requested, inventory-low, book-created).
- **Benefits**: Decoupled services, async agent workflows, proactive inventory restocking via event triggers
- **Effort**: Medium
- **Priority**: High — aligns with customer preference for EventBridge

**Opportunity 2: Async Return Processing**
- **Current State**: Return approval in local-monolith is a synchronous multi-step transaction. No async patterns in unishop-monolith.
- **Proposed Solution**: SQS-backed return processing workers on ECS. Agent submits return → receives immediate acknowledgment → polls for completion or receives EventBridge notification.
- **Benefits**: Non-blocking agent interactions, better scalability, error isolation
- **Effort**: Medium
- **Priority**: Medium — improves agent UX for return processing

### API Gateway Consolidation

- **Current State**: aws-microservices has 3 separate API Gateways (Product, Basket, Order). Other services have separate entry points.
- **Proposed Solution**: Consider a unified API Gateway per domain group with shared configuration (auth, throttling, logging). Alternatively, maintain per-service gateways but standardize configuration via CDK/Terraform modules.
- **Benefits**: Consistent auth, rate limiting, monitoring across all services. Single agent entry point with unified tooling.
- **Effort**: Medium
- **Priority**: Medium

### Observability Unification

- **Current State**: books-api has X-Ray tracing. aws-microservices, local-monolith, and unishop-monolith have no observability.
- **Proposed Solution**: Standardized observability stack — OpenTelemetry for tracing, Lambda Powertools/monolog for logging, CloudWatch for metrics. Shared dashboards with service-level and agent-level views.
- **Benefits**: End-to-end tracing from agent → all backend services, consistent structured logs, unified alerting
- **Effort**: Medium
- **Priority**: High

## Risk Analysis

### Technical Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| **No authentication on aws-microservices** — All 12 API endpoints publicly accessible. An agent with public APIs creates critical data breach risk. | High | High | Phase 1: Deploy Cognito authorizer on all API Gateway endpoints. Interim: add API key requirement immediately. |

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| **4 different programming languages** — PHP, Java 8, JavaScript, TypeScript creates skill fragmentation and limits cross-team mobility. | Medium | Medium | Build agent layer in Python or TypeScript (single language). Maintain existing backends. Standardize new services on TypeScript. |

### Dependency Risks

1. **Circular dependency (aws-microservices ↔ books-api)**: Prevents independent deployment and scaling. If one service fails, the other may cascade. **Must be resolved in Phase 0.**

### Single Points of Failure

1. **aws-microservices + books-api coupled pair**: Together they form a 50% blast radius unit. Failure in either service affects the other due to circular dependency.
2. **unishop-monolith on single EC2**: No auto-scaling, no multi-AZ, no failover. Single instance failure takes the entire service offline.
3. **local-monolith's single-AZ RDS**: `db.t3.micro` with no Multi-AZ failover. Database failure takes the entire order management system offline.
