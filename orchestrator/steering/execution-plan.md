# Execution Plan Generation

How to run the Execution Plan (EBA) TD after ARA and MODA analyses are complete. Read this when the user wants to generate a dependency-aware execution roadmap from analysis findings.

> **The Execution Plan is generated by Kiro, not by ct.** It uses `atx custom def exec` with `additionalPlanContext` via `-g file://atx-config-exec-plan.yaml`. This is the ONE place where `additionalPlanContext` is still used and relevant — ct handles ARA/MODA with no custom context (built-in types reject `-g`), but Kiro handles EBA generation with full `additionalPlanContext`.
>
> **Kiro's role:** Dynamically generate `atx-config-exec-plan.yaml` from discovered repos and ct findings, then run the EBA TD.

---

## When to Run

- After **both** ARA and MODA analyses are complete (status: `complete`)
- The user asked for `analysis_type: full` (which includes EBA)
- The user explicitly requests an execution plan / roadmap

---

## Prerequisites

1. **ARA analysis complete**: `atx ct analysis list --type agentic-readiness` shows status `complete`
2. **MODA analysis complete**: `atx ct analysis list --type modernization-readiness` shows status `complete`
3. **Report artifacts available**: `atx ct analysis list-artifacts --id <ara-id>` returns artifacts
4. **No concurrent `atx ct analysis run` in progress** — the ct server and custom exec may conflict on git state

---

## Retrieving Artifacts for EBA Input

Since ct stores artifacts in its internal store (not on disk), you must export them before running EBA:

```bash
# Get the analysis IDs
ara_id=$(atx ct analysis list --type agentic-readiness --status complete --json | jq -r '.[0].id')
moda_id=$(atx ct analysis list --type modernization-readiness --status complete --json | jq -r '.[0].id')

# List available artifacts
atx ct analysis list-artifacts --id $ara_id --json
atx ct analysis list-artifacts --id $moda_id --json

# Export per-repo reports to local files
atx ct analysis get-artifact --id $ara_id --repo <source>::<repo-name> --name report > ./ara-reports/<repo-name>.json
atx ct analysis get-artifact --id $moda_id --repo <source>::<repo-name> --name report > ./moda-reports/<repo-name>.json
```

## EBA Config Generation (Interactive Workflow)

The key insight: `service_inventory[]`, findings, and dependencies come from ct automatically. The user only provides **execution constraints** — team capacity, timeline, budget, parallelism — that determine how the roadmap is sequenced and phased. An AI can't infer these from code.

### Step 1: Auto-populate from ct data (no user input needed)

Kiro gathers this automatically:

```bash
# Get repo list → service_inventory[].name, .path, .repo_type, .tags
atx ct repository list --source <name> --json

# Get findings per repo → severity distribution, categories
atx ct findings list --source <name> --json

# Dependencies → inferred from code analysis findings (cross-service call patterns)
atx ct findings list --source <name> --type agentic-readiness --json
```

**Auto-populated fields:**

| Config field | ct data source |
|---|---|
| `portfolio_name` | ct source name |
| `service_inventory[].name` | Repository slug (last segment) |
| `service_inventory[].path` | Repository local path from `repository list` |
| `service_inventory[].repo_type` | Inferred from findings categories |
| `service_inventory[].tags` | Repository labels from ct |
| `service_inventory[].findings_summary` | Severity counts from `findings list` |
| `dependency_overrides[]` | Inferred from cross-service findings (API calls, shared DBs) |

### Step 2: Prompt the user for execution constraints (interactive)

These are the HUMAN constraints that determine roadmap phasing. Ask conversationally — one or two at a time.

**Required:**

> "How many engineers or teams are available for this modernization effort?"

→ Maps to `team_size` / capacity constraint

> "What's your timeline? (e.g., 6 months, 12 months, 18 months)"

→ Maps to `timeline_constraint`

**Important (ask next):**

> "What's the budget for this effort? (include training, infrastructure, tooling)"

→ Maps to `budget_constraint`

> "How many services can you modernize simultaneously? (parallel workstream capacity)"

→ Maps to parallelism constraint (affects phasing)

**Optional (offer but don't require):**

> "Are there any hard deadlines or compliance dates? (e.g., SOC2 audit in March, PCI renewal in Q4)"

→ Maps to `compliance_requirements` / date constraints

> "Any services that MUST be done first due to business reasons? (regardless of technical priority)"

→ Maps to sequencing priority overrides

### Step 3: Generate the config

Combine ct data + execution constraints into `atx-config-exec-plan.yaml`:

```yaml
additionalPlanContext: |
  portfolio_name: "my-platform"
  team_size: 8
  timeline_constraint: "12 months"
  budget_constraint: "$1.2M including training and infrastructure"
  parallel_capacity: 3
  compliance_requirements: ["SOC2 audit by 2026-03", "PCI-DSS renewal Q4"]
  sequencing_overrides: ["payments-service must complete first"]
  service_inventory:
    - name: "payments-service"
      path: "/absolute/path/to/services/payments-service"
      priority: "P0"
      repo_type: "application"
      tags: ["java", "spring-boot"]
      findings_summary: {high: 4, medium: 12, low: 3}
    - name: "user-service"
      path: "/absolute/path/to/services/user-service"
      priority: "P1"
      repo_type: "application"
      tags: ["python", "fastapi"]
      findings_summary: {high: 1, medium: 8, low: 5}
  dependency_overrides:
    - source: "payments-service"
      target: "user-service"
      type: "sync"
      description: "REST API calls for user validation"
```

### Step 4: Show the user for confirmation

Before running EBA, present the generated config summary:

> "Here's the execution plan config:
>
> - **Portfolio:** my-platform (3 services, 47 total findings)
> - **Capacity:** 8 engineers, 3 parallel workstreams, 12 months
> - **Budget:** $1.2M
> - **Hard constraints:** SOC2 audit by March, PCI renewal Q4
> - **Sequencing:** payments-service first (business requirement)
> - **Dependencies:** payments-service → user-service (sync/REST)
>
> Want me to generate the execution plan with these constraints, or adjust anything?"

Only proceed to the execution command after user confirms.

### Configuration Fields Reference

| Field | Required | Source |
|---|---|---|
| `portfolio_name` | Yes | Auto-derived from ct source name |
| `team_size` | Yes | User input (Step 2) |
| `timeline_constraint` | Yes | User input (Step 2) |
| `budget_constraint` | No | User input (Step 2) |
| `parallel_capacity` | No | User input (Step 2) |
| `compliance_requirements` | No | User input (Step 2) |
| `sequencing_overrides` | No | User input (Step 2) |
| `service_inventory[]` | Yes | Auto-populated from ct (Step 1) |
| `dependency_overrides[]` | No | Auto-inferred from findings (Step 1) |

> **Critical:** `service_inventory[]` and `dependency_overrides[]` MUST be structured YAML object arrays — never free-text bullet lists. The EBA TD parses these programmatically.

### Why `additionalPlanContext` Is Used Here (and Nowhere Else)

- **ARA/MODA** (`atx ct analysis run`): Built-in types reject `-g`/`--configuration`. No custom context needed — ct handles everything.
- **EBA** (`atx custom def exec`): Requires execution constraints (capacity, budget, timeline) plus portfolio metadata to produce a phased roadmap. This is the ONLY place `additionalPlanContext` is relevant in this Power.

---

## Execution Command

```bash
atx custom def exec \
  -n portfolio-execution-plan-generation \
  -p . \
  -g file://atx-config-exec-plan.yaml \
  -x \
  -t
```

### Flags

| Flag | Purpose |
|---|---|
| `-n` | Transformation definition name |
| `-p .` | Workspace root (portfolio-level TD always uses `.`) |
| `-g file://...` | Configuration file with `file://` prefix |
| `-x` | Non-interactive (mandatory for automated runs) |
| `-t` | Trust all tools (mandatory for automated runs) |

### Always Use Absolute Paths in Automated Flows

In subagent/automated flows, compute absolute paths:

```bash
workspace_abs=$(pwd)
config_abs="$workspace_abs/atx-config-exec-plan.yaml"
atx custom def exec \
  -n portfolio-execution-plan-generation \
  -p "$workspace_abs" \
  -g "file://$config_abs" \
  -x -t
```

---

## Timeout

| TD type | timeout (ms) | Wall clock |
|---|---|---|
| Execution Plan | 1800000 | 30 minutes |

Use the `executeBash` tool's `timeout` parameter directly.

---

## Safety Contract

1. **Run ONLY after both ARA and MODA are complete** — verify with `atx ct analysis list`.
2. **Verify report artifacts are available** via `atx ct analysis list-artifacts`.
3. **Issue exactly one `executeBash` call** with timeout 1800000 ms. Do not poll, background, or split.
4. **Do NOT run concurrently with `atx ct analysis run`** — git state conflicts.
5. **After completion**, verify the execution plan artifact exists before reporting success.

---

## Expected Output

The EBA TD produces a **four-artifact bundle** in a `portfolio-execution-plan/` directory at the workspace root:

```
portfolio-execution-plan/
├── {portfolio-name}-portfolio-exec-plan.md           # Narrative roadmap
├── {portfolio-name}-portfolio-exec-plan.json         # Machine-readable plan (canonical)
├── {portfolio-name}-portfolio-exec-plan.html         # Self-contained HTML
└── {portfolio-name}-portfolio-exec-plan.metadata.json # Sidecar metadata
```

The JSON file contains:
- Prioritized execution phases and work streams
- Dependency-aware sequencing across services
- Per-task effort estimates and ownership
- Risk mitigation recommendations
- Decision points for stakeholder review
- Quick wins vs. long-term investments

### Verification

```bash
ls portfolio-execution-plan/*-portfolio-exec-plan.md
```

If the `.md` file exists, the EBA completed successfully.
