Executive Summary

Repository Status: 🟡 Pilot-Ready

Key Findings: This EKS SaaS GitOps infrastructure repository demonstrates strong compute and data platform maturity (EKS + Karpenter + DynamoDB) but has significant gaps in observability, incident response, and security audit controls. The GitOps deployment model via Flux CD is well-architected but lacks progressive delivery and quality gates.

Remediation Plan: Priority actions focus on adding distributed tracing (ADOT/OpenTelemetry), CloudTrail audit logging, CI pipeline quality gates (security scanning, terraform validate), and integration testing for the tenant lifecycle.

Recommended Actions: (1) Deploy ADOT for distributed tracing, (2) Add CloudTrail with immutable logs, (3) Add security scanning to CI pipeline.

Detailed Findings

MEDIUM OPS-Q1: Distributed Tracing

Category: Operations & Observability

Finding

No distributed tracing (X-Ray, OpenTelemetry, Jaeger) found in any IaC, Helm chart, or GitOps manifest. Complete absence in a multi-tenant, multi-service architecture.

Recommendation

Deploy ADOT as DaemonSet on EKS. Instrument microservices with OpenTelemetry SDK. Enable X-Ray for SQS/DynamoDB tracing.

MEDIUM OPS-Q3: Business Metrics

Category: Operations & Observability

Finding

No custom business metrics published. Kubecost tracks costs but no tenant onboarding rates, message throughput, or per-tier utilization metrics exist.

Recommendation

Add custom Prometheus metrics in microservices (messages_processed_total, onboarding_duration_seconds). Create Grafana dashboards for business KPIs.

MEDIUM SEC-Q1: Audit Logging

Category: Security Baseline

Finding

No CloudTrail configuration despite provisioning account-level resources (EKS, VPC, IAM). EKS control plane logging not enabled.

Recommendation

Add CloudTrail with log file validation and S3 Object Lock. Enable EKS control plane logging (api, audit, authenticator).

MEDIUM SEC-Q4: Centralized Identity

Category: Security Baseline

Finding

No centralized IdP integration. Gitea uses local auth. EKS uses IAM. No SSO or federated identity configuration.

Recommendation

Integrate Amazon Cognito for tenant apps. Federate EKS access with organizational IdP via OIDC.

MEDIUM SEC-Q6: Compute Hardening

Category: Security Baseline

Finding

Karpenter uses al2023@latest AMI with SSM policy. No Patch Manager, vulnerability scanning, or hardened AMI pipeline.

Recommendation

Enable ECR scanning. Add Trivy to CI. Configure SSM Patch Manager. Consider Bottlerocket AMIs.

MEDIUM OPS-Q2: SLO Definitions

Category: Operations & Observability

Finding

Prometheus deployed via Kubecost but no formal SLO definitions, no latency/error-rate alarms, no error budgets.

Recommendation

Define SLOs for tenant onboarding latency, message processing, and API availability per tier.

MEDIUM OPS-Q4: Anomaly Detection

Category: Operations & Observability

Finding

Prometheus deployed but no alerting rules, no anomaly detection, no PagerDuty/OpsGenie integration configured.

Recommendation

Define PrometheusRule CRDs. Add Alertmanager with notification integration. Add CloudWatch anomaly detection.

MEDIUM OPS-Q6: Integration Testing

Category: Operations & Observability

Finding

Helm test and manual scripts exist but no automated integration tests in CI pipeline. No E2E testing of tenant lifecycle.

Recommendation

Create integration tests for tenant onboarding workflow. Add to CI or scheduled Argo Workflow.

MEDIUM OPS-Q7: Incident Response

Category: Operations & Observability

Finding

No runbooks, SSM Automation, or self-healing patterns defined despite Argo Workflows being available.

Recommendation

Create runbooks for common scenarios. Implement self-healing via Argo Workflows triggered by Prometheus alerts.

MEDIUM OPS-Q8: Observability Ownership

Category: Operations & Observability

Finding

Kubecost provides namespace-level cost tracking but no per-service dashboards, named alarm owners, or CODEOWNERS.

Recommendation

Define CODEOWNERS for observability configs. Create per-tier Grafana dashboards with ownership annotations.

MEDIUM OPS-Q9: Resource Tagging

Category: Operations & Observability

Finding

Basic tagging (Blueprint, Name) but no consistent strategy. Missing cost allocation, ownership, and environment tags.

Recommendation

Add default_tags to AWS provider. Add required-tags Config rule. Implement Tag Policies.

LOW INF-Q3: Workflow Orchestration

Category: Infrastructure & DevOps

Finding

Argo Workflows for tenant lifecycle automation but no application-level business workflow orchestration.

Recommendation

Consider Step Functions or extend Argo Workflows for business-critical multi-step operations.

LOW INF-Q5: Network Security

Category: Infrastructure & DevOps

Finding

VPC with 3 AZs and private subnets but EKS endpoint is public and no VPC endpoints configured.

Recommendation

Restrict EKS endpoint to private. Add VPC endpoints for ECR, STS, S3, DynamoDB, SQS.

LOW INF-Q6: API Entry Point

Category: Infrastructure & DevOps

Finding

ALB via AWS LB Controller but no API Gateway, throttling, or request validation at entry point.

Recommendation

Add AWS WAF on ALB. Evaluate API Gateway for per-tenant throttling.

LOW INF-Q7: Auto-Scaling

Category: Infrastructure & DevOps

Finding

Karpenter provides node scaling but HPA disabled by default in application Helm charts.

Recommendation

Enable HPA by default with sensible CPU/memory targets. Add custom metrics for scaling.

LOW INF-Q8: Backup and Recovery

Category: Infrastructure & DevOps

Finding

DynamoDB PITR enabled but no documented restore procedures and no Gitea backup strategy.

Recommendation

Add S3 versioning. Implement Gitea backup. Document restore procedures.

LOW INF-Q9: High Availability

Category: Infrastructure & DevOps

Finding

3 AZ deployment but single NAT Gateway creates SPOF. Gitea is single-instance.

Recommendation

Deploy NAT Gateways per AZ. Evaluate managed Git service or HA Gitea.
