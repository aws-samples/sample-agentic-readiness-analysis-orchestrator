Findings

AUTH-Q1: No Machine Identity Authentication BLOCKER

Description: Single shared API key for all programmatic access. No service accounts, client credentials OAuth 2.0, mTLS, or distinct machine identity support.

Gap: Cannot distinguish which agent or service made a call. Cannot attribute API calls to individual agent identities.

Recommendation: Implement multiple API keys with distinct identity labels, or add OAuth 2.0 client credentials via API gateway.

Evidence: src/Sonarr.Http/Authentication/ApiKeyAuthenticationHandler.cs

AUTH-Q2: No Scoped Permissions RISK-SAFETY

Description: Binary access model — valid API key grants full access to all operations. No RBAC or scoped permissions.

Gap: Cannot restrict an agent to read-only access on specific resources.

Recommendation: Implement RBAC or scoped API key permissions.

AUTH-Q3: No Action-Level Authorization RISK-SAFETY

Description: No differentiation between read, create, update, and delete at the authorization layer.

Gap: Cannot restrict read vs write per resource type.

Recommendation: Add action-level authorization attributes or ABAC model.

AUTH-Q6: No Immutable Audit Logging RISK-SAFETY

Description: Logs do not record authenticated principal identity. No immutable or tamper-evident log storage.

Gap: Cannot attribute actions to specific identities.

Recommendation: Add principal attribution to logging middleware. Configure immutable log storage.

AUTH-Q7: No Agent Identity Suspension RISK-SAFETY

Description: Single API key — cannot suspend individual agent without affecting all consumers.

Gap: No per-identity suspension capability.

Recommendation: Implement multi-key support with per-key lifecycle management.

STATE-Q4: No Circuit Breakers RISK-SAFETY

Description: Polly retry only. No circuit breaker pattern for external dependency calls.

Gap: Agent requests could cascade failures through external dependencies.

Recommendation: Add Polly CircuitBreaker policies on external HTTP calls.

STATE-Q5: No Inbound Rate Limiting RISK-SAFETY

Description: No inbound API rate limiting. Only outbound throttling exists.

Gap: No protection against agent-speed inbound traffic.

Recommendation: Add ASP.NET Core rate limiting middleware or gateway-level limits.

DATA-Q1: Sensitive Credentials in API Responses RISK-SAFETY

Description: HostConfigResource API returns API key, hashed password, proxy credentials, and SSL cert password without filtering.

Gap: Sensitive credentials exposed in general API responses.

Recommendation: Add [JsonIgnore] to sensitive fields. Create separate privileged endpoints for credentials.

DATA-Q6: Incomplete PII Redaction in Logs RISK-SAFETY

Description: Log sanitization covers API keys/passwords but not broader PII (IP addresses, file paths).

Gap: Incomplete PII redaction.

Recommendation: Extend sanitization patterns to cover all PII categories.

API-Q2: OpenAPI Spec Lacks Contract Validation RISK-QUALITY

Description: OpenAPI specs exist and are CI-maintained, but no contract testing validates accuracy.

Recommendation: Add OpenAPI diff and contract validation to CI.

API-Q3: No Retryable Indicator in Errors RISK-QUALITY

Description: Structured error bodies lack retryable boolean or error category.

Recommendation: Add retryable field to error response format.
