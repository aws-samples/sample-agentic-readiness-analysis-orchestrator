Executive Summary

Repository Status: 🟠 Remediation Required

Key Findings: Netflix Eureka is a Java 8 service-discovery server and client built with legacy Netflix OSS stack (Jersey 1.x, Servlet 2.5, AWS SDK v1). Detailed Findings

INF-Q1: No managed compute — WAR deployment with no containerizationHIGH
Category: Infrastructure & DevOps
Finding

All compute is undefined. Application packaged as WAR for external servlet container. No ECS, EKS, Lambda, Fargate, or EC2 resource definitions exist.

Recommendation

Containerize: convert WAR to embedded server, create Dockerfile, deploy to EKS. Define EKS cluster and service resources in Terraform or CDK.

INF-Q5: No network security infrastructure definedHIGH
Category: Infrastructure & DevOps
Finding

No VPC, subnet, security group, or network policy definitions exist. Zero network security configuration.

Recommendation

Define VPC with private subnets, security groups restricting access to port 8080, and VPC endpoints for AWS service access.

INF-Q10: Zero Infrastructure as CodeHIGH
Category: Infrastructure & DevOps
Finding

No Terraform, CloudFormation, CDK, Helm charts, or Kustomize files exist. All infrastructure must be manually created.

Recommendation

Add Terraform or CDK for EKS cluster, VPC, ALB, ECR, IAM, CloudWatch alarms, and Route 53 records.

SEC-Q1: No CloudTrail or audit loggingHIGH
Category: Security Baseline
Finding

No CloudTrail configured. Application logging via Log4j 1.x (EOL) with console appender only. No structured logs, no retention.

Recommendation

Configure CloudTrail for API-level auditing. Replace Log4j 1.x with Logback + JSON encoder. Ship to CloudWatch Logs.

OPS-Q5: No deployment strategyHIGH
Category: Operations & Observability
Finding

Repository only publishes library artifacts to Maven Central. No production deployment pipeline, no canary, no blue/green.

Recommendation

Implement rolling updates with health probes in EKS. For higher safety, add canary deployments via Argo Rollouts.

OPS-Q6: Minimal integration testingHIGH
Category: Operations & Observability
Finding

Single integration test among ~110 primarily unit test files. No peer replication or multi-instance cluster tests.

Recommendation

Add Testcontainers-based integration suite for multi-instance peer replication, client failover, and registry lifecycle.

INF-Q3: Multi-step operations hardcoded with no orchestrationMEDIUM
Category: Infrastructure & DevOps
Finding

Peer replication batching, registry sync, and self-preservation are hardcoded with custom batching frameworks.

Recommendation

Evaluate Step Functions for peer replication observability. Current in-process approach may be appropriate for latency-sensitive service discovery.

INF-Q4: No messaging — peer replication via synchronous HTTPMEDIUM
Category: Infrastructure & DevOps
Finding

No messaging infrastructure. All peer replication via synchronous HTTP batch requests. No SQS, SNS, EventBridge, or any messaging.

Recommendation

Evaluate EventBridge or SQS for peer state propagation. Migrating to AWS Cloud Map eliminates this concern.

INF-Q11: CI/CD limited to artifact publicationMEDIUM
Category: Infrastructure & DevOps
Finding

GitHub Actions automate build and publish to Maven Central, but no deployment pipeline exists for production workloads.

Recommendation

Extend with Docker build, ECR push, Helm packaging, and EKS deployment. Add IaC pipeline.

APP-Q1: Compound legacy — Java 8, Jersey 1.x, AWS SDK v1MEDIUM
Category: Application Architecture
Finding

Java 8 with Jersey 1.x (1.19.1), AWS SDK v1 (1.11.277), Servlet API 2.5. Compound regression across all axes.

Recommendation

Migrate to Java 17+ with Spring Boot 3.x and AWS SDK v2. Sequence: Java 17 → Spring Boot 3.x → AWS SDK v2.

APP-Q2: Monolith with shared in-memory stateMEDIUM
Category: Application Architecture
Finding

10 Gradle submodules with clear boundaries, but deployed as single WAR. Shared in-memory registry couples all modules.

Recommendation

Containerize as single service first. Evaluate AWS Cloud Map as managed replacement.

🟠 Modernization Recommendations

  1. Containerize the application — Convert WAR to embedded server, create Dockerfile, deploy to EKS with proper VPC networking and security groups (addresses INF-Q1, INF-Q5, INF-Q10).
  2. Establish CI/CD deployment pipeline — Extend GitHub Actions with container build, ECR push, and EKS deployment with canary strategy (addresses INF-Q11, OPS-Q5).
  3. Add security foundations — Configure CloudTrail, replace Log4j 1.x, add Dependabot for vulnerability scanning, implement mTLS for peer communication (addresses SEC-Q1, SEC-Q6, SEC-Q7).
