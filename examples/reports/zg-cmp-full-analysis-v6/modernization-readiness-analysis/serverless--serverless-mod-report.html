Detailed Findings

HIGH SEC-Q1: No Audit Logging Configuration

Category: Security Baseline
Finding
No CloudTrail or equivalent audit logging configuration exists in the repository. Release operations to S3/CloudFront across 3 AWS accounts have no audit trail configuration defined as code.
Recommendation
Define CloudTrail configuration for the release AWS accounts as IaC. Ensure log file validation is enabled and logs are stored in an immutable S3 bucket with Object Lock.

MEDIUM APP-Q5: No Formal API Versioning Strategy

Category: Application Architecture
Finding
The CLI uses semantic versioning for releases but has no formal API versioning for the plugin interface, MCP server tool definitions, or serverless.yml configuration schema.
Recommendation
Add explicit API versioning to the MCP server tool definitions. Document the plugin API contract with version markers.

MEDIUM SEC-Q3: MCP Server Lacks Per-Request Authentication

Category: Security Baseline
Finding
The MCP server exposes HTTP endpoints without per-request authentication, relying solely on local execution trust. CLI uses AWS SSO/OIDC flows correctly.
Recommendation
Add token-based authentication to the MCP HTTP server. Consider API key validation or bearer token checks for the Express endpoints.

MEDIUM SEC-Q4: Platform Auth Uses Proprietary Keys

Category: Security Baseline
Finding
AWS SSO integration works well, but the Serverless Inc platform uses its own license key and access key system independent of any centralized IdP.
Recommendation
Support standard OIDC/OAuth2 flows for platform authentication alongside existing license key mechanism to enable SSO for enterprise users.

MEDIUM SEC-Q7: No SAST in CI/CD Pipeline

Category: Security Baseline
Finding
Dependabot provides dependency scanning. Snyk-bot is referenced but no SAST workflow step exists in the CI pipeline. No container scanning for 12 Dockerfiles.
Recommendation
Add CodeQL or Semgrep SAST scanning as a GitHub Actions workflow. Add container scanning for Dockerfiles. Configure security gates.

MEDIUM OPS-Q4: No Anomaly Detection or Alerting

Category: Operations & Observability
Finding
Axiom integration provides data collection but no alert definitions exist. No alerting on release failures or CLI error rate degradation.
Recommendation
Configure alerts in Axiom for release pipeline failures, abnormal error rates, and significant usage drops indicating broken releases.

MEDIUM OPS-Q7: No Incident Response Automation or Runbooks

Category: Operations & Observability
Finding
No runbooks, incident response automation, or self-healing patterns found. No structured incident response process.
Recommendation
Create structured runbooks for failed releases, broken canary deployments, auth issues, and npm publish failures. Automate rollback via workflow_dispatch.

MEDIUM OPS-Q8: No Explicit Observability Ownership

Category: Operations & Observability
Finding
Observability exists but no CODEOWNERS file, no named alarm owners, no team attribution on dashboards or telemetry assets.
Recommendation
Add CODEOWNERS file with explicit ownership for observability code paths. Document team ownership for Axiom datasets.

MEDIUM OPS-Q9: No Resource Tagging Governance

Category: Operations & Observability
Finding
No resource tagging standards or enforcement. Release workflows interact with S3/CloudFront but no tags are applied. No default tags for user-deployed resources.
Recommendation
Add default tags to framework-managed AWS resources in release scripts. Add default_tags configuration option to serverless.yml.

LOW APP-Q1: AWS SDK v2 Legacy Dependency Present

Category: Application Architecture
Finding
AWS SDK v3 is the primary SDK but legacy AWS SDK v2 (^2.1693.0) remains in the serverless package alongside v3.
Recommendation
Complete migration from AWS SDK v2 to v3. Replace the aws-sdk dependency with specific v3 client packages.

LOW APP-Q6: External Service Endpoints Not Fully Centralized

Category: Application Architecture
Finding
External service endpoints use environment variables with some hard-coded fallback URLs scattered across source files.
Recommendation
Centralize external service endpoint configuration in a single configuration module.

LOW DATA-Q2: AWS SDK Client Usage Not Fully Centralized

Category: Data Platform
Finding
The resolver pattern centralizes data access well, but individual plugins create their own SDK clients without a shared factory.
Recommendation
Centralize AWS SDK client creation through a shared client factory in the util or sf-core package.
