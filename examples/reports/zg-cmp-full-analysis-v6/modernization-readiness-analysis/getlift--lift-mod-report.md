### Application Architecture

#### APP-Q1: Programming Languages

| Field | Value |
|-------|-------|
| **Score** | 3 |
| **Finding** | The library is written in TypeScript 4.3 targeting ES2019, with Node.js 16.6.2 specified in `.nvmrc`. The CI matrix tests on Node 20, 22, and 24 (modern). The `aws-cdk-lib` ^2.215.0 dependency provides deep AWS SDK v2/v3 interop. However, TypeScript 4.x is now outdated (TypeScript 5.x is current), and the `aws-sdk` v2 is used in devDependencies for testing. |
| **Gap** | TypeScript 4.3 is behind current TypeScript 5.x. The `.nvmrc` specifies Node 16 (approaching EOL), though CI tests on modern Node versions. `aws-sdk` v2 is used in dev/test code. |
| **Recommendation** | Upgrade TypeScript to 5.x for improved type inference and performance. Update `.nvmrc` to Node 20 LTS. Migrate test code from `aws-sdk` v2 to `@aws-sdk` v3 client packages. |
| **Evidence** | `package.json` (typescript ^4.3.4, aws-sdk ^2.1322.0), `.nvmrc` (v16.6.2), `.github/workflows/ci.yml` (matrix: node 20, 22, 24) |

| Field | Value |

Each construct is an independent class with its own schema and lifecycle. The provider pattern acts as a factory with no circular dependencies between constructs. |

#### APP-Q5: API Versioning Strategy

| Field | Value |
|-------|-------|
| **Score** | 2 |
| **Finding** | The library publishes as an npm package (`serverless-lift`) with semver versioning for releases. However, there is no explicit versioning strategy for the construct schemas (the public API surface). The `CONSTRUCTS_DEFINITION` and individual construct schemas (`QUEUE_DEFINITION`, `STORAGE_DEFINITION`, etc.) define the public contract but have no migration path or deprecation mechanism when schema shapes change. The `peerDependencies` field declares `serverless: ^3 || ^4` compatibility. |
| **Gap** | No deprecation warnings, schema migration utilities, or versioned construct types exist. Breaking changes to construct configuration (e.g., changing `STORAGE_DEFINITION` properties) would silently break existing `serverless.yml` files. No CHANGELOG or breaking change documentation process is visible. |
| **Recommendation** | Introduce construct schema versioning (e.g., `type: "queue/v2"` alongside `type: "queue"`) or implement a deprecation-warning mechanism that alerts users when they use about-to-change configuration properties. Add a CHANGELOG.md with explicit breaking-change callouts per release. |
| **Evidence** | `src/constructs/aws/Queue.ts` (QUEUE_DEFINITION), `src/constructs/aws/Storage.ts` (STORAGE_DEFINITION), `package.json` (peerDependencies, semver) |

#### APP-Q6: Service Discovery

| Field | Value |

Constructs are located by type string, and the plugin resolves construct variables via the `${construct:id.property}` variable resolver. Cross-construct references are handled through CDK Tokens and Serverless Framework variable resolution. |

#### DATA-Q4: Stored Procedures and Schema Complexity

| Field | Value |
|-------|-------|
| **Score** | 2 |
| **Finding** | The `DatabaseDynamoDBSingleTable` construct hard-codes a specific single-table design pattern with fixed attribute names (`PK`, `SK`), up to 5 Local Secondary Indexes (with hard-coded `LSI{n}SK` sort keys), and configurable Global Secondary Indexes (with `GSI{n}PK`/`GSI{n}SK` keys). While DynamoDB has no stored procedures, this rigid schema definition couples consumers to a specific data modeling approach that may not fit all use cases. The `TimeToLive` attribute is hard-coded. |
| **Gap** | The hard-coded schema (PK/SK attribute names, LSI naming convention, TTL attribute name) is opinionated and inflexible. Consumers who need different attribute names, composite sort keys, or different TTL field names cannot customize without forking. This is analogous to database-level coupling via proprietary constructs. |
| **Recommendation** | Make the DynamoDB construct's attribute names configurable via the schema definition. Allow consumers to specify custom partition/sort key names, TTL attribute name, and LSI/GSI sort key patterns. Keep the current defaults as sensible out-of-box behavior. |
| **Evidence** | `src/constructs/aws/DatabaseDynamoDBSingleTable.ts` (hard-coded `PK`, `SK`, `TimeToLive`, `LSI{n}SK`, `GSI{n}PK/SK`) |
