Detailed Findings

DATA-Q3: Dependency version ranges and Node.js EOL risk MEDIUM
Category: Data Platform
Finding

All 11 AWS SDK dependencies use caret ranges (^3.750.0) allowing minor version drift. Node.js 18 engine minimum is approaching/past EOL. CI matrix includes non-LTS Node 21.

Recommendation

Update minimum Node.js engine to >=20 (current LTS). Add Node.js 22 to CI matrix and remove 21 (non-LTS). Document version update procedure for AWS SDK dependencies.

APP-Q5: No formal API versioning strategy MEDIUM
Category: Application Architecture
Finding

The library uses npm semver (v6.2.2) but has no CHANGELOG, no deprecation annotations, and README still references AWS SDK v2 patterns despite being rewritten for v3.

Recommendation

Add a CHANGELOG.md documenting breaking changes per version. Add JSDoc @deprecated annotations for legacy APIs. Update README to accurately reflect v3-only status.

SEC-Q3: No input validation on public API functions MEDIUM
Category: Security Baseline
Finding

setSDK() uses require(path) with a user-provided path without validation. setSDKInstance() accepts any object without type checking. While this is a test utility, lack of validation could enable loading arbitrary modules.

Recommendation

Add input validation to setSDK() to restrict the path parameter. Document that setSDK() should only be used with trusted paths in test environments.

SEC-Q7: No SAST in CI pipeline LOW
Category: Security Baseline
Finding

Dependabot is configured for dependency scanning. ESLint is configured but not run in CI (lint script runs only tsc). No SAST tool (CodeQL, Semgrep) is integrated.

Recommendation

Add GitHub CodeQL or Semgrep to CI workflow for SAST coverage. Fix lint script to include ESLint alongside TypeScript compilation.

SEC-Q6: Dependency vulnerability scanning gaps LOW
Category: Security Baseline
Finding

Dependabot handles dependency updates but no explicit npm audit step exists in CI to block builds on critical vulnerabilities.

Recommendation

Add npm audit --audit-level=high to the CI workflow to fail builds on known high-severity vulnerabilities in dependencies.

APP-Q2: Single-file module structure limits maintainability LOW
Category: Application Architecture
Finding

All core logic resides in a single 664-line file (src/index.ts) rather than being decomposed into focused internal modules.

Recommendation

Consider internal decomposition of src/index.ts into focused modules (registry.ts, service-mocker.ts, method-mocker.ts) for maintainability.

APP-Q6: Limited default client registry LOW
Category: Application Architecture
Finding

Default registry covers only 10 AWS services. No auto-discovery of installed @aws-sdk/client-* packages.

Recommendation

Consider implementing automatic discovery of installed @aws-sdk/client-* packages to reduce manual configuration.

DATA-Q1: No structured documentation generation pipeline LOW
Category: Data Platform
Finding

Documentation is manually maintained markdown. No automated TypeDoc or similar generation pipeline exists.

Recommendation

Add TypeDoc generation to the build pipeline for structured API documentation from JSDoc comments.

OPS-Q1: No trace context consideration in mocked responses LOW
Category: Operations & Observability
Finding

No OpenTelemetry instrumentation. Mocked AWS calls do not propagate trace context (expected for mocks but undocumented).

Recommendation

Document that mocked calls do not propagate trace context so consumers know to use real clients for tracing tests.

Modernization Recommendation

Top recommendations based on assessment findings:

  1. Update Node.js lifecycle: Bump minimum engine to >=20 LTS, update CI matrix to test Node 20 and 22
  2. Formalize API versioning: Add CHANGELOG.md, deprecation annotations, and update README for v3-only status
  3. Strengthen security pipeline: Add SAST scanning (CodeQL/Semgrep) and npm audit to CI workflow
