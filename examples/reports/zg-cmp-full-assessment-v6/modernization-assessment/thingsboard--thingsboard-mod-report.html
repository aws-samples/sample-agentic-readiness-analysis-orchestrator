Executive Summary

Repository Status: 🟠 Remediation Required

Key Findings: ThingsBoard is a well-structured IoT platform with modern Java 25 / Spring Boot 3.5.12 stack and existing microservices deployment capability via Docker Compose + Kafka. However, it lacks cloud infrastructure fundamentals: no IaC, no CI/CD pipelines, no managed compute, no encryption at rest, and no audit logging. The platform has strong application architecture foundations (modular design, pluggable services, comprehensive test suite) but requires significant infrastructure modernization to operate reliably on AWS.

Remediation Plan: 4 modernization pathways triggered — Move to Cloud Native (decompose shared database), Move to Managed Databases (Aurora PostgreSQL, ElastiCache), Move to Managed Analytics (MSK Serverless replacing self-managed Kafka), and Move to Modern DevOps (IaC + CI/CD from scratch).

Recommended Actions: (1) Establish IaC with AWS CDK for EKS + Aurora + MSK; (2) Build CI/CD pipeline in GitHub Actions; (3) Enable encryption at rest and audit logging as security baseline.

Detailed Findings

HIGHINF-Q10: Zero infrastructure-as-code

Category: Infrastructure & DevOps

Finding

No Terraform, CloudFormation, CDK, Helm, Kustomize, or Ansible. All infrastructure defined via Docker Compose files only. 100% of cloud infrastructure would need manual creation.

Recommendation

Adopt AWS CDK to define EKS cluster, Aurora PostgreSQL, MSK, ElastiCache, VPC networking. Create Helm charts for service deployments.

HIGHINF-Q1: No managed compute

Category: Infrastructure & DevOps

Finding

All compute runs on Docker Compose with HAProxy. 13 Dockerfiles define services but no managed container orchestration (ECS, EKS, Fargate) is used.

Recommendation

Migrate to Amazon EKS with Helm charts for each service type. Use managed node groups with Graviton instances.

HIGHINF-Q5: No network security infrastructure

Category: Infrastructure & DevOps

Finding

No VPC, subnet, security group, or network segmentation. Services on flat Docker network with no isolation between data, application, and transport tiers.

Recommendation

Deploy in VPC with private subnets. Use security groups with least-privilege. Place API Gateway in front of HTTP endpoints.

HIGHINF-Q11: No CI/CD pipeline

Category: Infrastructure & DevOps

Finding

Three GitHub Actions workflows exist but none perform build, test, or deployment. Strong test infrastructure (128 black-box tests, JUnit 5) not leveraged in any automated pipeline.

Recommendation

Implement GitHub Actions CI/CD: Maven build → unit tests → Docker image build/push to ECR → integration tests → deploy to EKS.

HIGHSEC-Q1: No infrastructure audit logging

Category: Security Baseline

Finding

No CloudTrail or equivalent. Application has entity-level audit log but no infrastructure-level audit trail or immutable log storage.

Recommendation

Enable AWS CloudTrail with log file validation and S3 Object Lock. Configure CloudWatch Logs for application log aggregation.

HIGHSEC-Q2: No encryption at rest

Category: Security Baseline

Finding

No encryption at rest for any data store. Device credentials, API keys, telemetry data, and tenant information stored unencrypted on Docker volumes.

Recommendation

Migrate to Aurora PostgreSQL (encryption at rest by default). Enable EBS encryption for EKS nodes. Use customer-managed KMS keys.

HIGHOPS-Q2: No SLO definitions

Category: Operations & Observability

Finding

No SLO definitions, no error budget tracking, no formal availability or latency targets for any user journey.

Recommendation

Define SLOs: MQTT connection success (99.9%), telemetry ingestion p99 (<500ms), REST API p99 (<200ms). Implement CloudWatch SLO monitoring.

HIGHOPS-Q5: No deployment strategy

Category: Operations & Observability

Finding

No blue/green, canary, or rolling deployment. Updates via docker-compose restart. Bad deployments affect all users immediately with no rollback.

Recommendation

Implement canary deployments on EKS using ArgoCD Rollouts or App Mesh. Define health check criteria for automatic promotion/rollback.

MEDIUMAPP-Q2: Modular monolith with shared database coupling

Category: Application Architecture

Finding

Modular monolith with identifiable modules that deploys as microservices via Kafka. Shared PostgreSQL with 982-line unified schema accessed by all modules prevents true independent deployment.

Recommendation

Apply database-per-service pattern incrementally. Start with transport services. Extract telemetry into DynamoDB. Extract alarm management.

MEDIUMINF-Q2: Databases self-managed in Docker

Category: Infrastructure & DevOps

Finding

PostgreSQL, Cassandra, and Redis/Valkey all run as Docker containers with no managed service, automated backups, failover, or patching.

Recommendation

Migrate to Amazon Aurora PostgreSQL, ElastiCache for Redis, and evaluate Timestream or DynamoDB for time-series.

MEDIUMSEC-Q5: Secrets in environment variables without rotation

Category: Security Baseline

Finding

All secrets externalized via environment variables. Docker env files contain default passwords (postgres, thingsboardDefaultSigningKey). No Secrets Manager integration or rotation.

Recommendation

Integrate AWS Secrets Manager. Configure automated rotation for database credentials. Use EKS Secrets Store CSI driver.

🟠 Modernization Recommendations

Establish infrastructure-as-code with AWS CDK defining EKS, Aurora PostgreSQL, MSK Serverless, and VPC networking — this is the foundation for all other modernization.
Implement CI/CD pipeline (GitHub Actions) with build, test, container image push to ECR, and deployment to EKS — enables safe, automated delivery.
Enable security baseline: CloudTrail audit logging, encryption at rest via Aurora/KMS, and Secrets Manager for credential rotation.
