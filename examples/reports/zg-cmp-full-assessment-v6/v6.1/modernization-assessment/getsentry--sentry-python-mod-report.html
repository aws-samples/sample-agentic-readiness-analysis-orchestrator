Detailed Findings

MEDIUM APP-Q5: No formal API versioning strategy beyond package semver

Category: Application Architecture
Finding
The SDK uses semantic versioning for package releases and has a MIGRATION_GUIDE.md, but lacks explicit public API surface declarations, stability tiers, or deprecation lifecycle policy.
Recommendation
Define explicit API stability tiers. Add a formal deprecation policy. Use __all__ consistently or a @public decorator to formalize the API surface.

MEDIUM SEC-Q4: No centralized IdP integration for CI/CD workflows

Category: Security Baseline
Finding
CI/CD and release pipelines use GitHub-native auth mechanisms (GitHub App tokens, repository secrets). No centralized enterprise IdP federation.
Recommendation
For an open-source library, GitHub-native auth is standard. Document how DSN authentication maps to enterprise IAM patterns for enterprise consumers.

MEDIUM SEC-Q6: CI runners use default images with no hardening

Category: Security Baseline
Finding
CI workflows run on GitHub-hosted runners with default images. SHA-pinned actions provide supply-chain security, but no custom hardening or vulnerability scanning of CI environment.
Recommendation
Add container scanning for the Python 3.6 Docker container used in CI. Consider verifying runner image provenance.

LOW APP-Q1: Python SDK uses EOL minimum version (3.6) and legacy build system

Category: Application Architecture
Finding
The SDK supports Python 3.6-3.14t with modern tooling (ruff, mypy strict) but minimum Python 3.6 is past EOL and build uses legacy setup.py.
Recommendation
Drop Python 3.6-3.7 support. Migrate build system from setup.py to pyproject.toml with a modern build backend.

LOW APP-Q3: Some integration paths use synchronous blocking patterns

Category: Application Architecture
Finding
Background worker handles async event submission effectively, but some integrations use synchronous blocking, limited by Python 3.6 minimum support.
Recommendation
As Python 3.6-3.7 support is dropped, migrate remaining synchronous transport paths to native async where applicable.

LOW SEC-Q5: Secrets managed via GitHub Actions but no rotation policy

Category: Security Baseline
Finding
No plaintext credentials in the repository. All secrets managed through GitHub Actions encrypted secrets. No documented rotation policy.
Recommendation
Document a secrets rotation schedule. Use OIDC-based trusted publishers for all external service authentication where supported.

LOW SEC-Q7: No SAST tool in CI pipeline

Category: Security Baseline
Finding
Dependabot and FOSSA provide dependency scanning and license compliance. No SAST for code-level security vulnerabilities.
Recommendation
Add Bandit or Semgrep with Python security rules to the CI pipeline as a required check.

LOW OPS-Q1: SDK lacks internal telemetry for its own operations

Category: Operations & Observability
Finding
The SDK implements distributed tracing for consuming applications but does not instrument its own internal operations for observability.
Recommendation
Add optional internal SDK telemetry that consumers can enable to monitor SDK health.

LOW APP-Q6: Integration auto-discovery lacks machine-readable catalog

Category: Application Architecture
Finding
The SDK uses plugin-based auto-discovery via import-time detection and entry_points but has no machine-readable catalog of integrations with version compatibility.
Recommendation
Publish a machine-readable integration catalog (JSON/YAML) documenting each integration's name, supported library versions, and feature matrix.

LOW DATA-Q2: Integration data capture patterns are inconsistent

Category: Data Platform
Finding
The SDK has a centralized data pipeline but 65+ integration modules use varying patterns for data extraction (monkey-patching, hooks, decorators).
Recommendation
Define a formal Integration Data Protocol standardizing how integrations capture spans, breadcrumbs, and context.

LOW DATA-Q3: Database integration minimums include EOL library versions

Category: Data Platform
Finding
The SDK supports database drivers with minimum versions including EOL releases: SQLAlchemy 1.2 (past EOL), pymongo 3.1 (legacy).
Recommendation
Align minimum supported database driver versions with actively maintained versions. Bump minimums to sqlalchemy>=1.4, pymongo>=4.0.

LOW DATA-Q4: SQL query capture lacks portable vs proprietary classification

Category: Data Platform
Finding
The SDK captures raw SQL through database integrations but does not distinguish between portable SQL and proprietary/stored-procedure SQL in span descriptions.
Recommendation
Consider adding query classification metadata to database spans indicating portable vs proprietary SQL patterns.

LOW SEC-Q3: DSN authentication uses static keys without rotation

Category: Security Baseline
Finding
The SDK authenticates to Sentry backend using DSN-based static public keys with no support for scoped or short-lived tokens.
Recommendation
Consider supporting scoped, short-lived authentication tokens as an alternative to static DSNs for stricter environments.
